# PostgreSQL

PostgreSQL（常简称为 Postgres）是功能完善的开源关系型数据库，支持标准 SQL、事务、MVCC、丰富的数据类型以及 JSONB 等扩展能力。

## 基础

### SQL 分类

- `DDL`（Data Definition Language，数据定义语言）：定义数据库对象，例如 `CREATE`、`ALTER`、`DROP`。
- `DML`（Data Manipulation Language，数据操作语言）：增、删、改数据，例如 `INSERT`、`UPDATE`、`DELETE`。
- `DQL`（Data Query Language，数据查询语言）：查询数据，核心关键字为 `SELECT`。
- `DCL`（Data Control Language，数据控制语言）：管理角色和权限，例如 `GRANT`、`REVOKE`。
- `TCL`（Transaction Control Language，事务控制语言）：控制事务，例如 `BEGIN`、`COMMIT`、`ROLLBACK`。

### 常用命令

```sql
-- 连接后列出数据库、切换数据库（psql 元命令）
\l
\c app_db

CREATE DATABASE app_db;
CREATE ROLE app_user LOGIN PASSWORD 'replace-with-a-secure-password';
GRANT ALL PRIVILEGES ON DATABASE app_db TO app_user;
```

> **注意**：`\l`、`\c` 等以反斜杠开头的命令属于 `psql` 客户端，不是 SQL；应用程序通过连接字符串指定目标数据库，不能像 MySQL 那样使用 `USE database` 切换。

### 标识符与数据类型

- 未加双引号的标识符会被折叠为小写；推荐统一使用小写的 `snake_case`。`"User"` 与 `user` 是不同标识符，且前者此后必须始终带双引号。
- 常用数值类型：`smallint`、`integer`、`bigint`、`numeric(p, s)`；金额应使用 `numeric`，避免浮点误差。
- 常用文本类型：`text`、`varchar(n)`、`char(n)`。除长度限制外，`text` 与 `varchar` 的性能通常没有本质差异。
- 常用时间类型：`date`、`time`、`timestamp`、`timestamptz`。业务时间通常优先使用带时区的 `timestamptz`。
- PostgreSQL 还原生支持 `uuid`、数组、范围类型、`jsonb` 等。

## 查询、约束与设计

### 查询

```sql
SELECT id, username, created_at
FROM users
WHERE status = 'active'
  AND created_at >= DATE '2026-01-01'
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;
```

PostgreSQL 支持 `RETURNING`，可在写入后直接返回受影响的记录，避免额外查询：

```sql
INSERT INTO users (username, email)
VALUES ('alice', 'alice@example.com')
RETURNING id, created_at;

UPDATE users
SET status = 'inactive'
WHERE id = 42
RETURNING id, status;
```

### 约束

约束用于保证数据的正确性、有效性和完整性。常用约束包括：

1. 主键：`PRIMARY KEY`，非空且唯一。
2. 非空：`NOT NULL`。
3. 唯一：`UNIQUE`。
4. 外键：`FOREIGN KEY`。
5. 检查：`CHECK`，限制列值或行内字段关系。
6. 默认值：`DEFAULT`，为未提供值的列填入默认值。

```sql
CREATE TABLE departments (
    id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL UNIQUE,
    created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE employees (
    id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    department_id bigint NOT NULL,
    name text NOT NULL,
    email text NOT NULL,
    salary numeric(12, 2) NOT NULL CHECK (salary >= 0),
    status text NOT NULL DEFAULT 'active'
        CHECK (status IN ('active', 'inactive')),
    CONSTRAINT uq_employees_email UNIQUE (email),
    CONSTRAINT fk_employees_department
        FOREIGN KEY (department_id)
        REFERENCES departments (id)
        ON UPDATE CASCADE
        ON DELETE RESTRICT
);
```

#### 自增主键

推荐使用 SQL 标准的 identity 列：

```sql
CREATE TABLE orders (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    order_no text NOT NULL UNIQUE
);
```

- `GENERATED ALWAYS AS IDENTITY`：默认不允许手动指定主键值。
- `GENERATED BY DEFAULT AS IDENTITY`：允许在导入历史数据等场景手动指定值。
- `serial` / `bigserial` 是由序列和默认值组成的历史简写；新表优先使用 identity。

#### 修改与删除约束

PostgreSQL 通常需要通过约束名称删除约束；可在创建时显式命名以便维护。

```sql
ALTER TABLE employees
    ADD CONSTRAINT ck_employees_name_not_blank CHECK (length(trim(name)) > 0);

ALTER TABLE employees
    ALTER COLUMN name SET NOT NULL;

ALTER TABLE employees
    DROP CONSTRAINT uq_employees_email;
```

#### 外键与级联操作

外键维护表之间的引用完整性。删除或更新被引用记录时，常见行为如下：

- `RESTRICT` / `NO ACTION`：存在引用记录时拒绝操作（默认通常为 `NO ACTION`）。
- `CASCADE`：级联更新或删除引用记录。
- `SET NULL`：将外键设为 `NULL`，外键列必须允许为空。
- `SET DEFAULT`：将外键设为默认值。

多表关系的建模方式与其他关系型数据库一致：一对多在“多”的一方保存外键；多对多使用中间表并为两列设置联合主键；一对一可在任一方使用唯一外键。

```sql
CREATE TABLE student_courses (
    student_id bigint REFERENCES students (id) ON DELETE CASCADE,
    course_id bigint REFERENCES courses (id) ON DELETE CASCADE,
    enrolled_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (student_id, course_id)
);
```

### 数据库设计范式

- 第一范式（1NF）：每一列都是不可分割的原子值。
- 第二范式（2NF）：在 1NF 基础上，非主属性完全依赖于整个候选键，消除部分依赖。
- 第三范式（3NF）：在 2NF 基础上，非主属性不依赖于其他非主属性，消除传递依赖。

范式降低冗余和更新异常，但实际设计还要结合查询模式。有意识地反范式化时，应明确数据来源与同步策略。

## 多表与事务

### 多表查询

```sql
-- 显式内连接
SELECT e.name AS employee_name, d.name AS department_name
FROM employees AS e
JOIN departments AS d ON d.id = e.department_id;

-- 左外连接：保留没有员工的部门
SELECT d.name, e.name
FROM departments AS d
LEFT JOIN employees AS e ON e.department_id = d.id;

-- 公共表表达式（CTE）
WITH department_counts AS (
    SELECT department_id, count(*) AS employee_count
    FROM employees
    GROUP BY department_id
)
SELECT d.name, coalesce(c.employee_count, 0) AS employee_count
FROM departments AS d
LEFT JOIN department_counts AS c ON c.department_id = d.id;
```

### 事务

PostgreSQL 的每一条独立 SQL 语句默认都在一个事务中自动提交。使用 `BEGIN` 显式开启事务后，必须以 `COMMIT` 或 `ROLLBACK` 结束。

```sql
CREATE TABLE accounts (
    id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL UNIQUE,
    balance numeric(12, 2) NOT NULL CHECK (balance >= 0)
);

-- 转账：任一语句失败时，回滚整个事务
BEGIN;

UPDATE accounts
SET balance = balance - 500
WHERE name = 'zhangsan' AND balance >= 500;

-- 应用应检查上一条 UPDATE 的影响行数；为 0 时执行 ROLLBACK。
UPDATE accounts
SET balance = balance + 500
WHERE name = 'lisi';

COMMIT;
```

发生错误后，当前事务会进入失败状态；在执行 `ROLLBACK`（或回滚到保存点）前，后续 SQL 不会执行。

```sql
BEGIN;
SAVEPOINT before_discount;
UPDATE orders SET discount = 100 WHERE id = 1;
-- 发现折扣不符合业务规则
ROLLBACK TO SAVEPOINT before_discount;
COMMIT;
```

#### ACID

- 原子性（Atomicity）：事务中的操作全部成功，或全部回滚。
- 一致性（Consistency）：事务前后均满足约束和业务不变量。
- 隔离性（Isolation）：并发事务不会暴露不应看到的中间状态。
- 持久性（Durability）：已提交事务在故障后仍可恢复。

#### 隔离级别与 MVCC

PostgreSQL 通过 MVCC（多版本并发控制）让读操作通常不阻塞写操作。支持以下隔离级别：

| 隔离级别 | PostgreSQL 行为 |
| --- | --- |
| `READ UNCOMMITTED` | 按 `READ COMMITTED` 执行，不会产生脏读。 |
| `READ COMMITTED`（默认） | 每条语句获取新的已提交数据快照。 |
| `REPEATABLE READ` | 整个事务使用一致快照；遇到并发更新可能报序列化错误。 |
| `SERIALIZABLE` | 提供可串行化效果；应用必须准备好重试序列化失败的事务。 |

对需要“先读后写”的关键记录，可使用行级锁：

```sql
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

## 索引

索引能加速查询，但会增加写入开销和存储空间。为常出现在 `WHERE`、`JOIN`、`ORDER BY` 中且选择性较好的列建立索引，并使用执行计划验证效果。

```sql
-- 默认 B-tree 索引：等值、范围和排序查询的首选
CREATE INDEX idx_employees_department_id ON employees (department_id);

-- 联合索引遵循最左前缀原则
CREATE INDEX idx_employees_department_status
    ON employees (department_id, status);

-- 部分索引：仅索引活跃员工
CREATE INDEX idx_employees_active_email
    ON employees (email)
    WHERE status = 'active';

-- 检查实际执行计划；ANALYZE 会执行查询，生产环境需谨慎使用
EXPLAIN ANALYZE
SELECT * FROM employees
WHERE department_id = 10 AND status = 'active';
```

常见索引类型：

- B-tree：默认类型，适用于比较、范围和排序。
- GIN：适用于 `jsonb`、数组、全文检索等一对多键场景。
- GiST：适用于几何、范围、相似度等可扩展查询。
- BRIN：适用于数据与物理存储位置高度相关的大表时间列，索引很小。

```sql
CREATE TABLE events (
    id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    payload jsonb NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_events_payload_gin ON events USING gin (payload);

-- 查询 JSONB 中包含指定键值的事件
SELECT * FROM events
WHERE payload @> '{"event_type": "login"}'::jsonb;
```

## 常见差异：MySQL 与 PostgreSQL

| 场景 | MySQL 常见写法 | PostgreSQL 写法 |
| --- | --- | --- |
| 自增主键 | `AUTO_INCREMENT` | `GENERATED ... AS IDENTITY` |
| 当前时间 | `NOW()` | `now()` 或 `CURRENT_TIMESTAMP` |
| 布尔值 | 常见 `TINYINT(1)` | 原生 `boolean`，值为 `true` / `false` |
| 字符串拼接 | `CONCAT(a, b)` | `a || b`（也支持 `concat`） |
| 限制结果 | `LIMIT offset, count` | `LIMIT count OFFSET offset` |
| UPSERT | `ON DUPLICATE KEY UPDATE` | `ON CONFLICT (...) DO UPDATE` |

```sql
INSERT INTO departments (name)
VALUES ('研发部')
ON CONFLICT (name)
DO UPDATE SET name = EXCLUDED.name
RETURNING id, name;
```
